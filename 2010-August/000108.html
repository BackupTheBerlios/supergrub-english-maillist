Mapping and chainloading like this won't actually work with the vast
majority of live isos, you'll get a boot menu but the OS will fail to
load properly as it will look for its files on the CDROM drive
(accessing the hardware directly, not through calls to the BIOS).

I don't know why the author of this guide decided to do this when
isodetect.lua comes with SG2D and actually works with many popular
GNU/Linux isos. The only isos that I can think of that this method
will support are Parted Magic (which hard codes the file path to the
iso, so if you change the name of the iso it fails to boot) and
specially made Windows isos using winvblock (which if I understand
correctly uses BIOS calls even after the kernel is loaded). Parted
magic is already supported by isodetect.lua, without the limitation
that the file name stay the same. I haven't added support for Windows
isos with winvblock because, among other things, "map --hook" only
works with BIOS based systems, where isodetect.lua currently also
works with EFI, Coreboot, Open Firmware, etc.
